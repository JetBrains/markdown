tests - A quick guide</a></p></li></ul></li></ol><p>##Overview</p><p>The apache module allows you to set up virtual hosts and manage web services with minimal effort.</p><p>##Module Description</p><p>Apache is a widely-used web server, and this module provides a simplified way of creating configurations to manage your infrastructure. This includes the ability to configure and manage a range of different virtual host setups, as well as a streamlined way to install and configure Apache modules.</p><p>##Setup</p><p><strong>What apache affects:</strong></p><ul><li><p>configuration files and directories (created and written to)</p><ul><li><p><strong>WARNING</strong>: Configurations that are <emph>not</emph> managed by Puppet will be purged.</p></li></ul></li><li><p>package/service/configuration files for Apache</p></li><li><p>Apache modules</p></li><li><p>virtual hosts</p></li><li><p>listened-to ports</p></li><li><p><code>/etc/make.conf</code> on FreeBSD and Gentoo</p></li><li><p>depends on module 'gentoo/puppet-portage' for Gentoo</p></li></ul><p>###Beginning with Apache</p><p>To install Apache with the default parameters</p><pre><code>    class { 'apache':  }</code></pre><p>The defaults are determined by your operating system (e.g. Debian systems have one set of defaults, and RedHat systems have another, as do FreeBSD and Gentoo systems). These defaults work well in a testing environment, but are not suggested for production. To establish customized parameters</p><pre><code>    class { 'apache':
      default_mods        =&gt; false,
      default_confd_files =&gt; false,
    }</code></pre><p>###Configure a virtual host</p><p>Declaring the <code>apache</code> class creates a default virtual host by setting up a vhost on port 80, listening on all interfaces and serving <code>$apache::docroot</code>.</p><pre><code>    class { 'apache': }</code></pre><p>To configure a very basic, name-based virtual host</p><pre><code>    apache::vhost { 'first.example.com':
      port    =&gt; '80',
      docroot =&gt; '/var/www/first',
    }</code></pre><p><emph>Note:</emph> The default priority is 15. If nothing matches this priority, the alphabetically first name-based vhost is used. This is also true if you pass a higher priority and no names match anything else.</p><p>A slightly more complicated example, changes the docroot owner/group from the default 'root'</p><pre><code>    apache::vhost { 'second.example.com':
      port          =&gt; '80',
      docroot       =&gt; '/var/www/second',
      docroot_owner =&gt; 'third',
      docroot_group =&gt; 'third',
    }</code></pre><p>To set up a virtual host with SSL and default SSL certificates</p><pre><code>    apache::vhost { 'ssl.example.com':
      port    =&gt; '443',
      docroot =&gt; '/var/www/ssl',
      ssl     =&gt; true,
    }</code></pre><p>To set up a virtual host with SSL and specific SSL certificates</p><pre><code>    apache::vhost { 'fourth.example.com':
      port     =&gt; '443',
      docroot  =&gt; '/var/www/fourth',
      ssl      =&gt; true,
      ssl_cert =&gt; '/etc/ssl/fourth.example.com.cert',
      ssl_key  =&gt; '/etc/ssl/fourth.example.com.key',
    }</code></pre><p>Virtual hosts listen on '*' by default. To listen on a specific IP address</p><pre><code>    apache::vhost { 'subdomain.example.com':
      ip      =&gt; '127.0.0.1',
      port    =&gt; '80',
      docroot =&gt; '/var/www/subdomain',
    }</code></pre><p>To set up a virtual host with a wildcard alias for the subdomain mapped to a same-named directory, for example: <code>http://example.com.loc</code> to <code>/var/www/example.com</code></p><pre><code>    apache::vhost { 'subdomain.loc':
      vhost_name       =&gt; '*',
      port             =&gt; '80',
      virtual_docroot  =&gt; '/var/www/%-2+',
      docroot          =&gt; '/var/www',
      serveraliases    =&gt; ['*.loc',],
    }</code></pre><p>To set up a virtual host with suPHP</p><pre><code>    apache::vhost { 'suphp.example.com':
      port                =&gt; '80',
      docroot             =&gt; '/home/appuser/myphpapp',
      suphp_addhandler    =&gt; 'x-httpd-php',
      suphp_engine        =&gt; 'on',
      suphp_configpath    =&gt; '/etc/php5/apache2',
      directories         =&gt; { path =&gt; '/home/appuser/myphpapp',
        'suphp'           =&gt; { user =&gt; 'myappuser', group =&gt; 'myappgroup' },
      }
    }</code></pre><p>To set up a virtual host with WSGI</p><pre><code>    apache::vhost { 'wsgi.example.com':
      port                        =&gt; '80',
      docroot                     =&gt; '/var/www/pythonapp',
      wsgi_application_group      =&gt; '%{GLOBAL}',
      wsgi_daemon_process         =&gt; 'wsgi',
      wsgi_daemon_process_options =&gt; {
        processes    =&gt; '2',
        threads      =&gt; '15',
        display-name =&gt; '%{GROUP}',
       },
      wsgi_import_script          =&gt; '/var/www/demo.wsgi',
      wsgi_import_script_options  =&gt;
        { process-group =&gt; 'wsgi', application-group =&gt; '%{GLOBAL}' },
      wsgi_process_group          =&gt; 'wsgi',
      wsgi_script_aliases         =&gt; { '/' =&gt; '/var/www/demo.wsgi' },
    }</code></pre><p>Starting in Apache 2.2.16, HTTPD supports <a href="https://httpd.apache.org/docs/current/mod/mod_dir.html#fallbackresource">FallbackResource</a>, a simple replacement for common RewriteRules.</p><pre><code>    apache::vhost { 'wordpress.example.com':
      port                =&gt; '80',
      docroot             =&gt; '/var/www/wordpress',
      fallbackresource    =&gt; '/index.php',
    }</code></pre><p>Please note that the 'disabled' argument to FallbackResource is only supported since Apache 2.2.24.</p><p>See a list of all <a href="#defined-type-apachevhost">virtual host parameters</a>. See an extensive list of <a href="#virtual-host-examples">virtual host examples</a>.</p><p>##Usage</p><p>###Classes and Defined Types</p><p>This module modifies Apache configuration files and directories and purges any configuration not managed by Puppet. Configuration of Apache should be managed by Puppet, as non-Puppet configuration files can cause unexpected failures.</p><p>It is possible to temporarily disable full Puppet management by setting the <a href="#purge_configs"><code>purge_configs</code></a> parameter within the base <code>apache</code> class to 'false'. This option should only be used as a temporary means of saving and relocating customized configurations. See the <a href="#purge_configs"><code>purge_configs</code> parameter</a> for more information.</p><p>####Class: <code>apache</code></p><p>The apache module's primary class, <code>apache</code>, guides the basic setup of Apache on your system.</p><p>You can establish a default vhost in this class, the <code>vhost</code> class, or both. You can add additional vhost configurations for specific virtual hosts using a declaration of the <code>vhost</code> type.</p><p><strong>Parameters within <code>apache</code>:</strong></p><p>#####<code>allow_encoded_slashes</code></p><p>This sets the server default for the <a href="http://httpd.apache.org/docs/current/mod/core.html#allowencodedslashes"><code>AllowEncodedSlashes</code> declaration</a> which modifies the responses to URLs with <code></code> and <code>/</code> characters. The default is undefined, which omits the declaration from the server configuration and select the Apache default setting of <code>Off</code>. Allowed values are: <code>on</code>, <code>off</code> or <code>nodecode</code>.</p><p>#####<code>apache_version</code></p><p>Configures the behavior of the module templates, package names, and default mods by setting the Apache version. Default is determined by the class <code>apache::version</code> using the OS family and release. It should not be configured manually without special reason.</p><p>#####<code>conf_dir</code></p><p>Changes the location of the configuration directory the main configuration file is placed in. Defaults to '/etc/httpd/conf' on RedHat, '/etc/apache2' on Debian, '/usr/local/etc/apache22' on FreeBSD, and '/etc/apache2' on Gentoo.</p><p>#####<code>confd_dir</code></p><p>Changes the location of the configuration directory your custom configuration files are placed in. Defaults to '/etc/httpd/conf' on RedHat, '/etc/apache2/conf.d' on Debian, '/usr/local/etc/apache22' on FreeBSD, and '/etc/apache2/conf.d' on Gentoo.</p><p>#####<code>conf_template</code></p><p>Overrides the template used for the main apache configuration file. Defaults to 'apache/httpd.conf.erb'.</p><p><emph>Note:</emph> Using this parameter is potentially risky, as the module has been built for a minimal configuration file with the configuration primarily coming from conf.d/ entries.</p><p>#####<code>default_charset</code></p><p>If defined, the value will be set as <code>AddDefaultCharset</code> in the main configuration file. It is undefined by default.</p><p>#####<code>default_confd_files</code></p><p>Generates default set of include-able Apache configuration files under  <code>${apache::confd_dir}</code> directory. These configuration files correspond to what is usually installed with the Apache package on a given platform.</p><p>#####<code>default_mods</code></p><p>Sets up Apache with default settings based on your OS. Valid values are 'true', 'false', or an array of mod names.</p><p>Defaults to 'true', which includes the default <a href="https://github.com/puppetlabs/puppetlabs-apache/blob/master/manifests/default_mods.pp">HTTPD mods</a>.</p><p>If false, it only includes the mods required to make HTTPD work, and any other mods can be declared on their own.</p><p>If an array, the apache module includes the array of mods listed.</p><p>#####<code>default_ssl_ca</code></p><p>The default certificate authority, which is automatically set to 'undef'. This default works out of the box but must be updated with your specific certificate information before being used in production.</p><p>#####<code>default_ssl_cert</code></p><p>The default SSL certification, which is automatically set based on your operating system  ('/etc/pki/tls/certs/localhost.crt' for RedHat, '/etc/ssl/certs/ssl-cert-snakeoil.pem' for Debian, '/usr/local/etc/apache22/server.crt' for FreeBSD, and '/etc/ssl/apache2/server.crt' for Gentoo). This default works out of the box but must be updated with your specific certificate information before being used in production.</p><p>#####<code>default_ssl_chain</code></p><p>The default SSL chain, which is automatically set to 'undef'. This default works out of the box but must be updated with your specific certificate information before being used in production.</p><p>#####<code>default_ssl_crl</code></p><p>The default certificate revocation list to use, which is automatically set to 'undef'. This default works out of the box but must be updated with your specific certificate information before being used in production.</p><p>#####<code>default_ssl_crl_path</code></p><p>The default certificate revocation list path, which is automatically set to 'undef'. This default works out of the box but must be updated with your specific certificate information before being used in production.</p><p>#####<code>default_ssl_crl_check</code></p><p>Sets the default certificate revocation check level via the <a href="http://httpd.apache.org/docs/current/mod/mod_ssl.html#sslcarevocationcheck">SSLCARevocationCheck directive</a>, which is automatically set to 'undef'. This default works out of the box but must be specified when using CRLs in production. Only applicable to Apache 2.4 or higher, the value is ignored on older versions.</p><p>#####<code>default_ssl_key</code></p><p>The default SSL key, which is automatically set based on your operating system ('/etc/pki/tls/private/localhost.key' for RedHat, '/etc/ssl/private/ssl-cert-snakeoil.key' for Debian, '/usr/local/etc/apache22/server.key' for FreeBSD, and '/etc/ssl/apache2/server.key' for Gentoo). This default works out of the box but must be updated with your specific certificate information before being used in production.</p><p>#####<code>default_ssl_vhost</code></p><p>Sets up a default SSL virtual host. Defaults to 'false'. If set to 'true', sets up the following vhost:</p><pre><code>    apache::vhost { 'default-ssl':
      port            =&gt; 443,
      ssl             =&gt; true,
      docroot         =&gt; $docroot,
      scriptalias     =&gt; $scriptalias,
      serveradmin     =&gt; $serveradmin,
      access_log_file =&gt; &quot;ssl_${access_log_file}&quot;,
      }</code></pre><p>SSL vhosts only respond to HTTPS queries.</p><p>#####<code>default_type</code></p><p>(Apache httpd 2.2 only) MIME content-type that will be sent if the server cannot determine a type in any other way. This directive has been deprecated in Apache httpd 2.4, and only exists there for backwards compatibility of configuration files.</p><p>#####<code>default_vhost</code></p><p>Sets up a default virtual host. Defaults to 'true', set to 'false' to set up <a href="#configure-a-virtual-host">customized virtual hosts</a>.</p><p>#####<code>docroot</code></p><p>Changes the location of the default <a href="https://httpd.apache.org/docs/current/mod/core.html#documentroot">Documentroot</a>. Defaults to '/var/www/html' on RedHat, '/var/www' on Debian, '/usr/local/www/apache22/data' on FreeBSD, and '/var/www/localhost/htdocs' on Gentoo.</p><p>#####<code>error_documents</code></p><p>Enables custom error documents. Defaults to 'false'.</p><p>#####<code>group</code></p><p>Changes the group that Apache will answer requests as. The parent process will continue to be run as root, but resource accesses by child processes will be done under this group. By default, puppet will attempt to manage this group as a resource under <code>::apache</code>. If this is not what you want, set <a href="#manage_group"><code>manage_group</code></a> to 'false'. Defaults to the OS-specific default user for apache, as detected in <code>::apache::params</code>.</p><p>#####<code>httpd_dir</code></p><p>Changes the base location of the configuration directories used for the apache service. This is useful for specially repackaged HTTPD builds, but might have unintended consequences when used in combination with the default distribution packages. Defaults to '/etc/httpd' on RedHat, '/etc/apache2' on Debian, '/usr/local/etc/apache22' on FreeBSD, and '/etc/apache2' on Gentoo.</p><p>#####<code>keepalive</code></p><p>Enables persistent connections.</p><p>#####<code>keepalive_timeout</code></p><p>Sets the amount of time the server waits for subsequent requests on a persistent connection. Defaults to '15'.</p><p>#####<code>max_keepalive_requests</code></p><p>Sets the limit of the number of requests allowed per connection when KeepAlive is on. Defaults to '100'.</p><p>#####<code>lib_path</code></p><p>Specifies the location where apache module files are stored. It should not be configured manually without special reason.</p><p>#####<code>loadfile_name</code></p><p>Sets the file name for the module loadfile. Should be in the format *.load.  This can be used to set the module load order.</p><p>#####<code>log_level</code></p><p>Changes the verbosity level of the error log. Defaults to 'warn'. Valid values are 'emerg', 'alert', 'crit', 'error', 'warn', 'notice', 'info', or 'debug'.</p><p>#####<code>log_formats</code></p><p>Define additional <a href="https://httpd.apache.org/docs/current/mod/mod_log_config.html#logformat">LogFormats</a>. This is done in a Hash:</p><pre><code>  $log_formats = { vhost_common =&gt; '%v %h %l %u %t \%r\ %&gt;s %b' }</code></pre><p>#####<code>logroot</code></p><p>Changes the directory where Apache log files for the virtual host are placed. Defaults to '/var/log/httpd' on RedHat, '/var/log/apache2' on Debian, '/var/log/apache22' on FreeBSD, and '/var/log/apache2' on Gentoo.</p><p>#####<code>logroot_mode</code></p><p>Overrides the mode the default logroot directory is set to ($::apache::logroot). Defaults to undef. Do NOT give people write access to the directory the logs are stored
in without being aware of the consequences; see http://httpd.apache.org/docs/2.4/logs.html#security for details.</p><p>#####<code>manage_group</code></p><p>Setting this to 'false' stops the group resource from being created. This is for when you have a group, created from another Puppet module, you want to use to run Apache. Without this parameter, attempting to use a previously established group would result in a duplicate resource error.</p><p>#####<code>manage_user</code></p><p>Setting this to 'false' stops the user resource from being created. This is for instances when you have a user, created from another Puppet module, you want to use to run Apache. Without this parameter, attempting to use a previously established user would result in a duplicate resource error.</p><p>#####<code>mod_dir</code></p><p>Changes the location of the configuration directory your Apache modules configuration files are placed in. Defaults to '/etc/httpd/conf.d' for RedHat, '/etc/apache2/mods-available' for Debian, '/usr/local/etc/apache22/Modules' for FreeBSD, and '/etc/apache2/modules.d' on Gentoo.</p><p>#####<code>mpm_module</code></p><p>Determines which MPM is loaded and configured for the HTTPD process. Valid values are 'event', 'itk', 'peruser', 'prefork', 'worker', or 'false'. Defaults to 'prefork' on RedHat, FreeBSD and Gentoo, and 'worker' on Debian. Must be set to 'false' to explicitly declare the following classes with custom parameters:</p><ul><li><p><code>apache::mod::event</code></p></li><li><p><code>apache::mod::itk</code></p></li><li><p><code>apache::mod::peruser</code></p></li><li><p><code>apache::mod::prefork</code></p></li><li><p><code>apache::mod::worker</code></p></li></ul><p><emph>Note:</emph> Switching between different MPMs on FreeBSD is possible but quite difficult. Before changing <code>$mpm_module</code> you must uninstall all packages that depend on your currently-installed Apache.</p><p>#####<code>package_ensure</code></p><p>Allows control over the package ensure attribute. Can be 'present','absent', or a version string.</p><p>#####<code>ports_file</code></p><p>Changes the name of the file containing Apache ports configuration. Default is <code>${conf_dir}/ports.conf</code>.</p><p>#####<code>purge_configs</code></p><p>Removes all other Apache configs and vhosts, defaults to 'true'. Setting this to 'false' is a stopgap measure to allow the apache module to coexist with existing or otherwise-managed configuration. It is recommended that you move your configuration entirely to resources within this module.</p><p>#####<code>purge_vhost_configs</code></p><p>If <code>vhost_dir</code> != <code>confd_dir</code>, this controls the removal of any configurations that are not managed by Puppet within <code>vhost_dir</code>. It defaults to the value of <code>purge_configs</code>. Setting this to false is a stopgap measure to allow the apache module to coexist with existing or otherwise unmanaged configurations within <code>vhost_dir</code></p><p>#####<code>sendfile</code></p><p>Makes Apache use the Linux kernel sendfile to serve static files. Defaults to 'On'.</p><p>#####<code>serveradmin</code></p><p>Sets the server administrator. Defaults to 'root@localhost'.</p><p>#####<code>servername</code></p><p>Sets the server name. Defaults to <code>fqdn</code> provided by Facter.</p><p>#####<code>server_root</code></p><p>Sets the root directory in which the server resides. Defaults to '/etc/httpd' on RedHat, '/etc/apache2' on Debian, '/usr/local' on FreeBSD, and '/var/www' on Gentoo.</p><p>#####<code>server_signature</code></p><p>Configures a trailing footer line under server-generated documents. More information about <a href="http://httpd.apache.org/docs/current/mod/core.html#serversignature">ServerSignature</a>. Defaults to 'On'.</p><p>#####<code>server_tokens</code></p><p>Controls how much information Apache sends to the browser about itself and the operating system. More information about <a href="http://httpd.apache.org/docs/current/mod/core.html#servertokens">ServerTokens</a>. Defaults to 'OS'.</p><p>#####<code>service_enable</code></p><p>Determines whether the HTTPD service is enabled when the machine is booted. Defaults to 'true'.</p><p>#####<code>service_ensure</code></p><p>Determines whether the service should be running. Valid values are 'true', 'false', 'running', or 'stopped' when Puppet should manage the service. Any other value sets ensure to 'false' for the Apache service, which is useful when you want to let the service be managed by some other application like Pacemaker. Defaults to 'running'.</p><p>#####<code>service_name</code></p><p>Name of the Apache service to run. Defaults to: 'httpd' on RedHat, 'apache2' on Debian and Gentoo, and 'apache22' on FreeBSD.</p><p>#####<code>service_manage</code></p><p>Determines whether the HTTPD service state is managed by Puppet . Defaults to 'true'.</p><p>#####<code>service_restart</code></p><p>Determines whether the HTTPD service restart command should be anything other than the default managed by Puppet.  Defaults to undef.</p><p>#####<code>trace_enable</code></p><p>Controls how TRACE requests per RFC 2616 are handled. More information about <a href="http://httpd.apache.org/docs/current/mod/core.html#traceenable">TraceEnable</a>. Defaults to 'On'.</p><p>#####<code>vhost_dir</code></p><p>Changes the location of the configuration directory your virtual host configuration files are placed in. Defaults to 'etc/httpd/conf.d' on RedHat, '/etc/apache2/sites-available' on Debian, '/usr/local/etc/apache22/Vhosts' on FreeBSD, and '/etc/apache2/vhosts.d' on Gentoo.</p><p>#####<code>user</code></p><p>Changes the user that Apache will answer requests as. The parent process will continue to be run as root, but resource accesses by child processes will be done under this user. By default, puppet will attept to manage this user as a resource under <code>::apache</code>. If this is not what you want, set <a href="#manage_user"><code>manage_user</code></a> to 'false'. Defaults to the OS-specific default user for apache, as detected in <code>::apache::params</code>.</p><p>#####<code>apache_name</code></p><p>The name of the Apache package to install. This is automatically detected in <code>::apache::params</code>. You might need to override this if you are using a non-standard Apache package, such as those from Red Hat's software collections.</p><p>####Defined Type: <code>apache::custom_config</code></p><p>Allows you to create custom configs for Apache. The configuration files are only added to the Apache confd dir if the file is valid. An error is raised during the Puppet run if the file is invalid and <code>$verify_config</code> is <code>true</code>.</p><pre><code>    apache::custom_config { 'test':
        content =&gt; '# Test',

It is a hash, accepting <a href="http://modwsgi.readthedocs.org/en/latest/configuration-directives/WSGIDaemonProcess.html">these keys</a>, and it defaults to 'undef'.</p><p><code>wsgi_daemon_process_options</code> is optional and defaults to 'undef'.</p><p><code>wsgi_process_group</code> sets the group ID the virtual host runs under. Defaults to 'undef'.</p><p><code>wsgi_script_aliases</code> requires a hash of web paths to filesystem .wsgi paths. Defaults to 'undef'.</p><p><code>wsgi_pass_authorization</code> the WSGI application handles authorisation instead of Apache when set to 'On'. For more information see [here] (http://modwsgi.readthedocs.org/en/latest/configuration-directives/WSGIPassAuthorization.html). Defaults to 'undef'.</p><p>To set up a virtual host with WSGI</p><pre><code>    apache::vhost { 'wsgi.example.com':
      port                        =&gt; '80',
      docroot                     =&gt; '/var/www/pythonapp',
      wsgi_daemon_process         =&gt; 'wsgi',
      wsgi_daemon_process_options =&gt;
        { processes    =&gt; '2',
          threads      =&gt; '15',
          display-name =&gt; '%{GROUP}',
         },
      wsgi_process_group          =&gt; 'wsgi',
      wsgi_script_aliases         =&gt; { '/' =&gt; '/var/www/demo.wsgi' },
      wsgi_chunked_request        =&gt; 'On',
    }</code></pre><p>####Parameter <code>directories</code> for <code>apache::vhost</code></p><p>The <code>directories</code> parameter within the <code>apache::vhost</code> class passes an array of hashes to the vhost to create <a href="http://httpd.apache.org/docs/current/mod/core.html#directory">Directory</a>, <a href="http://httpd.apache.org/docs/current/mod/core.html#files">File</a>, and <a href="http://httpd.apache.org/docs/current/mod/core.html#location">Location</a> directive blocks. These blocks take the form, '&lt; Directory /path/to/directory&gt;...&lt; /Directory&gt;'.</p><p>The <code>path</code> key sets the path for the directory, files, and location blocks. Its value must be a path for the 'directory', 'files', and 'location' providers, or a regex for the 'directorymatch', 'filesmatch', or 'locationmatch' providers. Each hash passed to <code>directories</code> <strong>must</strong> contain <code>path</code> as one of the keys.</p><p>The <code>provider</code> key is optional. If missing, this key defaults to 'directory'. Valid values for <code>provider</code> are 'directory', 'files', 'location', 'directorymatch', 'filesmatch', or 'locationmatch'. If you set <code>provider</code> to 'directorymatch', it uses the keyword 'DirectoryMatch' in the Apache config file.</p><p>General <code>directories</code> usage looks something like</p><pre><code>    apache::vhost { 'files.example.net':
      docroot     =&gt; '/var/www/files',
      directories =&gt; [
        { 'path'     =&gt; '/var/www/files',
          'provider' =&gt; 'files',
          'deny'     =&gt; 'from all'
         },
      ],
    }</code></pre><p><emph>Note:</emph> At least one directory should match the <code>docroot</code> parameter. After you start declaring directories, <code>apache::vhost</code> assumes that all required Directory blocks will be declared. If not defined, a single default Directory block is created that matches the <code>docroot</code> parameter.</p><p>Available handlers, represented as keys, should be placed within the <code>directory</code>, <code>files</code>, or <code>location</code> hashes.  This looks like</p><pre><code>    apache::vhost { 'sample.example.net':

      ssl_verify_depth =&gt; 1,
    }</code></pre><p>#####<code>ssl_options</code></p><p>Sets the <a href="http://httpd.apache.org/docs/current/mod/mod_ssl.html#ssloptions">SSLOptions</a> directive, which configures various SSL engine run-time options. This is the global setting for the given vhost and can be a string or an array. Defaults to 'undef'.</p><p>A string:</p><pre><code>    apache::vhost { 'sample.example.net':
      
      ssl_options =&gt; '+ExportCertData',
    }</code></pre><p>An array:</p><pre><code>    apache::vhost { 'sample.example.net':
      
      ssl_options =&gt; [ '+StrictRequire', '+ExportCertData' ],
    }</code></pre><p>#####<code>ssl_proxyengine</code></p><p>Specifies whether or not to use <a href="http://httpd.apache.org/docs/current/mod/mod_ssl.html#sslproxyengine">SSLProxyEngine</a>. Valid values are 'true' and 'false'. Defaults to 'false'.</p><p>####Defined Type: FastCGI Server</p><p>This type is intended for use with mod_fastcgi. It allows you to define one or more external FastCGI servers to handle specific file types.</p><p>Ex:</p><pre><code>apache::fastcgi::server { 'php':
  host       =&gt; '127.0.0.1:9000',
  timeout    =&gt; 15,
  flush      =&gt; false,
  faux_path  =&gt; '/var/www/php.fcgi',
  fcgi_alias =&gt; '/php.fcgi',
  file_type  =&gt; 'application/x-httpd-php'
}</code></pre><p>Within your virtual host, you can then configure the specified file type to be handled by the fastcgi server specified above.</p><pre><code>apache::vhost { 'www':
  ...
  custom_fragment =&gt; 'AddType application/x-httpd-php .php'
  ...
}</code></pre><p>#####<code>host</code></p><p>The hostname or IP address and TCP port number (1-65535) of the FastCGI server.</p><p>#####<code>timeout</code></p><p>The number of seconds of FastCGI application inactivity allowed before the request is aborted and the event is logged (at the error LogLevel). The inactivity timer applies only as long as a connection is pending with the FastCGI application. If a request is queued to an application, but the application doesn't respond (by writing and flushing) within this period, the request is aborted. If communication is complete with the application but incomplete with the client (the response is buffered), the timeout does not apply.</p><p>#####<code>flush</code></p><p>Force a write to the client as data is received from the application. By default, mod_fastcgi buffers data in order to free the application as quickly as possible.</p><p>#####<code>faux_path</code></p><p><code>faux_path</code> does not have to exist in the local filesystem. URIs that Apache resolves to this filename are handled by this external FastCGI application.</p><p>#####<code>alias</code></p><p>A unique alias. This is used internally to link the action with the FastCGI server.</p><p>#####<code>file_type</code></p><p>The MIME-type of the file to be processed by the FastCGI server.</p><p>###Virtual Host Examples</p><p>The apache module allows you to set up pretty much any configuration of virtual host you might need. This section addresses some common configurations, but look at the <a href="https://github.com/puppetlabs/puppetlabs-apache/tree/master/tests">Tests section</a> for even more examples.</p><p>Configure a vhost with a server administrator</p><pre><code>    apache::vhost { 'third.example.com':
      port        =&gt; '80',
      docroot     =&gt; '/var/www/third',
      serveradmin =&gt; 'admin@example.com',
    }</code></pre><p>Set up a vhost with aliased servers</p><pre><code>    apache::vhost { 'sixth.example.com':
      serveraliases =&gt; [
        'sixth.example.org',
        'sixth.example.net',
      ],
      port          =&gt; '80',
      docroot       =&gt; '/var/www/fifth',
    }</code></pre><p>Configure a vhost with a cgi-bin</p><pre><code>    apache::vhost { 'eleventh.example.com':
      port        =&gt; '80',
      docroot     =&gt; '/var/www/eleventh',
      scriptalias =&gt; '/usr/lib/cgi-bin',
    }</code></pre><p>Set up a vhost with a rack configuration</p><pre><code>    apache::vhost { 'fifteenth.example.com':
      port           =&gt; '80',
      docroot        =&gt; '/var/www/fifteenth',
      rack_base_uris =&gt; ['/rackapp1', '/rackapp2'],
    }</code></pre><p>Set up a mix of SSL and non-SSL vhosts at the same domain</p><pre><code>    #The non-ssl vhost
    apache::vhost { 'first.example.com non-ssl':
      servername =&gt; 'first.example.com',
      port       =&gt; '80',
      docroot    =&gt; '/var/www/first',
    }

    #The SSL vhost at the same domain
    apache::vhost { 'first.example.com ssl':
      servername =&gt; 'first.example.com',
      port       =&gt; '443',
      docroot    =&gt; '/var/www/first',
      ssl        =&gt; true,
    }</code></pre><p>Configure a vhost to redirect non-SSL connections to SSL</p><pre><code>    apache::vhost { 'sixteenth.example.com non-ssl':
      servername      =&gt; 'sixteenth.example.com',
      port            =&gt; '80',
      docroot         =&gt; '/var/www/sixteenth',
      redirect_status =&gt; 'permanent',
      redirect_dest   =&gt; 'https://sixteenth.example.com/'
    }
    apache::vhost { 'sixteenth.example.com ssl':
      servername =&gt; 'sixteenth.example.com',
      port       =&gt; '443',
      docroot    =&gt; '/var/www/sixteenth',
      ssl        =&gt; true,
    }</code></pre><p>Set up IP-based vhosts on any listen port and have them respond to requests on specific IP addresses. In this example, we set listening on ports 80 and 81. This is required because the example vhosts are not declared with a port parameter.</p><pre><code>    apache::listen { '80': }
    apache::listen { '81': }</code></pre><p>Then we set up the IP-based vhosts</p><pre><code>    apache::vhost { 'first.example.com':
      ip       =&gt; '10.0.0.10',
      docroot  =&gt; '/var/www/first',
      ip_based =&gt; true,
    }
    apache::vhost { 'second.example.com':
      ip       =&gt; '10.0.0.11',
      docroot  =&gt; '/var/www/second',
      ip_based =&gt; true,
    }</code></pre><p>Configure a mix of name-based and IP-based vhosts. First, we add two IP-based vhosts on 10.0.0.10, one SSL and one non-SSL</p><pre><code>    apache::vhost { 'The first IP-based vhost, non-ssl':
      servername =&gt; 'first.example.com',
      ip         =&gt; '10.0.0.10',
      port       =&gt; '80',
      ip_based   =&gt; true,
      docroot    =&gt; '/var/www/first',
    }
    apache::vhost { 'The first IP-based vhost, ssl':
      servername =&gt; 'first.example.com',
      ip         =&gt; '10.0.0.10',
      port       =&gt; '443',
      ip_based   =&gt; true,
      docroot    =&gt; '/var/www/first-ssl',
      ssl        =&gt; true,
    }</code></pre><p>Then, we add two name-based vhosts listening on 10.0.0.20</p><pre><code>    apache::vhost { 'second.example.com':
      ip      =&gt; '10.0.0.20',
      port    =&gt; '80',
      docroot =&gt; '/var/www/second',
    }
    apache::vhost { 'third.example.com':
      ip      =&gt; '10.0.0.20',
      port    =&gt; '80',
      docroot =&gt; '/var/www/third',
    }</code></pre><p>If you want to add two name-based vhosts so that they answer on either 10.0.0.10 or 10.0.0.20, you <strong>MUST</strong> declare <code>add_listen =&gt; 'false'</code> to disable the otherwise automatic 'Listen 80', as it conflicts with the preceding IP-based vhosts.</p><pre><code>    apache::vhost { 'fourth.example.com':
      port       =&gt; '80',
      docroot    =&gt; '/var/www/fourth',
      add_listen =&gt; false,
    }
    apache::vhost { 'fifth.example.com':
      port       =&gt; '80',
      docroot    =&gt; '/var/www/fifth',
      add_listen =&gt; false,
    }</code></pre><p>###Load Balancing</p><p>####Defined Type: <code>apache::balancer</code></p><p><code>apache::balancer</code> creates an Apache balancer cluster. Each balancer cluster needs one or more balancer members, which are declared with <a href="#defined-type-apachebalancermember"><code>apache::balancermember</code></a>.</p><p>One <code>apache::balancer</code> defined resource should be defined for each Apache load balanced set of servers. The <code>apache::balancermember</code> resources for all balancer members can be exported and collected on a single Apache load balancer server using exported resources.</p><p><strong>Parameters within <code>apache::balancer</code>:</strong></p><p>#####<code>name</code></p><p>Sets the balancer cluster's title. This parameter also sets the title of the conf.d file.</p><p>#####<code>proxy_set</code></p><p>Configures key-value pairs as <a href="http://httpd.apache.org/docs/current/mod/mod_proxy.html#proxyset">ProxySet</a> lines. Accepts a hash, and defaults to '{}'.</p><p>#####<code>collect_exported</code></p><p>Determines whether or not to use exported resources. Valid values 'true' and 'false', defaults to 'true'.</p><p>If you statically declare all of your backend servers, you should set this to 'false' to rely on existing declared balancer member resources. Also make sure to use <code>apache::balancermember</code> with array arguments.</p><p>If you wish to dynamically declare your backend servers via <a href="http://docs.puppetlabs.com/guides/exported_resources.html">exported resources</a> collected on a central node, you must set this parameter to 'true' in order to collect the exported balancer member resources that were exported by the balancer member nodes.</p><p>If you choose not to use exported resources, all balancer members will be configured in a single Puppet run. If you are using exported resources, Puppet has to run on the balanced nodes, then run on the balancer.</p><p>####Defined Type: <code>apache::balancermember</code></p><p>Defines members of <a href="http://httpd.apache.org/docs/current/mod/mod_proxy_balancer.html">mod_proxy_balancer</a>, which sets up a balancer member inside a listening service configuration block in etc/apache/apache.cfg on the load balancer.</p><p><strong>Parameters within <code>apache::balancermember</code>:</strong></p><p>#####<code>name</code></p><p>Sets the title of the resource. This name also sets the name of the concat fragment.</p><p>#####<code>balancer_cluster</code></p><p>Sets the Apache service's instance name. This must match the name of a declared <code>apache::balancer</code> resource. Required.</p><p>#####<code>url</code></p><p>Specifies the URL used to contact the balancer member server. Defaults to 'http://${::fqdn}/'.</p><p>#####<code>options</code></p><p>An array of <a href="http://httpd.apache.org/docs/2.2/mod/mod_proxy.html#balancermember">options</a> to be specified after the URL. Accepts any key-value pairs available to <a href="http://httpd.apache.org/docs/2.2/mod/mod_proxy.html#proxypass">ProxyPass</a>.</p><p>####Examples</p><p>To load balance with exported resources, export the <code>balancermember</code> from the balancer member</p><pre><code>      @@apache::balancermember { &quot;${::fqdn}-puppet00&quot;:
        balancer_cluster =&gt; 'puppet00',
        url              =&gt; &quot;ajp://${::fqdn}:8009&quot;
        options          =&gt; ['ping=5', 'disablereuse=on', 'retry=5', 'ttl=120'],
      }</code></pre><p>Then, on the proxy server, create the balancer cluster</p><pre><code>      apache::balancer { 'puppet00': }</code></pre><p>To load balance without exported resources, declare the following on the proxy</p><pre><code>    apache::balancer { 'puppet00': }
    apache::balancermember { &quot;${::fqdn}-puppet00&quot;:
        balancer_cluster =&gt; 'puppet00',
        url              =&gt; &quot;ajp://${::fqdn}:8009&quot;
        options          =&gt; ['ping=5', 'disablereuse=on', 'retry=5', 'ttl=120'],
      }</code></pre><p>Then declare <code>apache::balancer</code> and <code>apache::balancermember</code> on the proxy server.</p><p>If you need to use ProxySet in the balancer config</p><pre><code>      apache::balancer { 'puppet01':
        proxy_set =&gt; {'stickysession' =&gt; 'JSESSIONID'},
      }</code></pre><p>##Reference</p><p>###Classes</p><p>####Public Classes</p><ul><li><p><a href="#class-apache"><code>apache</code></a>: Guides the basic setup of Apache.</p></li><li><p><code>apache::dev</code>: Installs Apache development libraries. (<emph>Note:</emph> On FreeBSD, you must declare <code>apache::package</code> or <code>apache</code> before <code>apache::dev</code>.)</p></li><li><p><a href="#classes-apachemodname"><code>apache::mod::[name]</code></a>: Enables specific Apache HTTPD modules.</p></li></ul><p>####Private Classes</p><ul><li><p><code>apache::confd::no_accf</code>: Creates the no-accf.conf configuration file in conf.d, required by FreeBSD's Apache 2.4.</p></li><li><p><code>apache::default_confd_files</code>: Includes conf.d files for FreeBSD.</p></li><li><p><code>apache::default_mods</code>: Installs the Apache modules required to run the default configuration.</p></li><li><p><code>apache::package</code>: Installs and configures basic Apache packages.</p></li><li><p><code>apache::params</code>: Manages Apache parameters.</p></li><li><p><code>apache::service</code>: Manages the Apache daemon.</p></li></ul><p>###Defined Types</p><p>####Public Defined Types</p><ul><li><p><code>apache::balancer</code>: Creates an Apache balancer cluster.</p></li><li><p><code>apache::balancermember</code>: Defines members of <a href="http://httpd.apache.org/docs/current/mod/mod_proxy_balancer.html">mod_proxy_balancer</a>.</p></li><li><p><code>apache::listen</code>: Based on the title, controls which ports Apache binds to for listening. Adds <a href="http://httpd.apache.org/docs/current/bind.html">Listen</a> directives to ports.conf in the Apache HTTPD configuration directory. Titles take the form '&lt;port&gt;', '&lt;ipv4&gt;:&lt;port&gt;', or '&lt;ipv6&gt;:&lt;port&gt;'.</p></li><li><p><code>apache::mod</code>: Used to enable arbitrary Apache HTTPD modules for which there is no specific <code>apache::mod::[name]</code> class.</p></li><li><p><code>apache::namevirtualhost</code>: Enables name-based hosting of a virtual host. Adds all <a href="http://httpd.apache.org/docs/current/vhosts/name-based.html">NameVirtualHost</a> directives to the <code>ports.conf</code> file in the Apache HTTPD configuration directory. Titles take the form '\', '*:&lt;port&gt;', '\default_:&lt;port&gt;, '&lt;ip&gt;', or '&lt;ip&gt;:&lt;port&gt;'.</p></li><li><p><code>apache::vhost</code>: Allows specialized configurations for virtual hosts that have requirements outside the defaults.</p></li></ul><p>####Private Defined Types</p><ul><li><p><code>apache::peruser::multiplexer</code>: Enables the <a href="http://www.freebsd.org/cgi/url.cgi?ports/www/apache22-peruser-mpm/pkg-descr">Peruser</a> module for FreeBSD only.</p></li><li><p><code>apache::peruser::processor</code>: Enables the <a href="http://www.freebsd.org/cgi/url.cgi?ports/www/apache22-peruser-mpm/pkg-descr">Peruser</a> module for FreeBSD only.</p></li><li><p><code>apache::security::file_link</code>: Links the activated_rules from apache::mod::security to the respective CRS rules on disk.</p></li></ul><p>###Templates</p><p>The Apache module relies heavily on templates to enable the <code>vhost</code> and <code>apache::mod</code> defined types. Unless explicitly called out, most templates are not meant for configuration.</p><p>##Limitations</p><p>###Ubuntu 10.04</p><p>The <code>apache::vhost::WSGIImportScript</code> parameter creates a statement inside the VirtualHost which is unsupported on older versions of Apache, causing this to fail.           command =&gt; 'semanage fcontext -a -t httpd_sys_content_t &quot;/custom/path(/.*)?&quot;',
          path    =&gt; '/bin:/usr/bin/:/sbin:/usr/sbin',
          require =&gt; Package['policycoreutils-python'],
        }
        package { 'policycoreutils-python': ensure =&gt; installed }
        exec { 'restorecon_apache':
          command =&gt; 'restorecon -Rv /apache_spec',
          path    =&gt; '/bin:/usr/bin/:/sbin:/usr/sbin',
          before  =&gt; Class['Apache::Service'],
          require =&gt; Class['apache'],
        }
        class { 'apache': }
        host { 'test.server': ip =&gt; '127.0.0.1' }
        file { '/custom/path': ensure =&gt; directory, }
        file { '/custom/path/include': ensure =&gt; present, content =&gt; '#additional_includes' }
        apache::vhost { 'test.server':
          docroot             =&gt; '/custom/path',
          additional_includes =&gt; '/custom/path/include',
        }</code></pre><p>You need to set the contexts using <code>semanage fcontext</code> not <code>chcon</code> because <code>file {...}</code> resources reset the context to the values in the database if the resource isn't specifying the context.</p><p>###FreeBSD</p><p>In order to use this module on FreeBSD, you <emph>must</emph> use apache24-2.4.12 (www/apache24) or newer.</p><p>##Development</p><p>###Contributing</p><p>Puppet Labs modules on the Puppet Forge are open projects, and community contributions are essential for keeping them great.
